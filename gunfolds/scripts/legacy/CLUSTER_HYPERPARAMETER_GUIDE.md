# Running Hyperparameter Tuning on SLURM Cluster

This guide explains how to run all 3125 priority configurations in parallel on your SLURM cluster.

## Overview

Instead of running 3125 configurations sequentially (which would take days), we run them **all in parallel** using SLURM array jobs. Each job tests one priority configuration across 5 batches.

## Files for Cluster Execution

1. **`VAR_hyperparameter_single_job.py`** - Python script that runs ONE priority config
2. **`slurm_hyperparameter_tuning.sh`** - SLURM submission script
3. **`VAR_collect_parallel_results.py`** - Collects all results after jobs complete

## Step-by-Step Instructions

### 1. Prepare Your Environment

Make sure your data files exist on the cluster:
```bash
~/DataSets_Feedbacks/8_VAR_simulation/net3/u3/txtSTD/data1.txt
~/DataSets_Feedbacks/8_VAR_simulation/net3/u3/txtSTD/data2.txt
...
~/DataSets_Feedbacks/8_VAR_simulation/net3/u3/txtSTD/data5.txt
```

### 2. Upload Scripts to Cluster

Copy these files to your cluster scripts directory:
```bash
scp VAR_hyperparameter_single_job.py mabavisani@cluster:/path/to/scripts/
scp slurm_hyperparameter_tuning.sh mabavisani@cluster:/path/to/scripts/
scp VAR_collect_parallel_results.py mabavisani@cluster:/path/to/scripts/
```

### 3. Configure SLURM Script (Optional)

Edit `slurm_hyperparameter_tuning.sh` if needed:

```bash
# Configuration parameters (lines 22-25)
NET=3                    # Change to your network number
UNDERSAMPLING=3          # Change to your undersampling rate
NUM_BATCHES=5           # Number of batches to average over
PNUM=1                  # CPUs per job (usually 1 for array jobs)
```

Also verify these match your cluster setup:
- `#SBATCH -p qTRDGPU` - Queue/partition name
- `#SBATCH -A psy53c17` - Account
- `#SBATCH --mem=2g` - Memory per job
- `#SBATCH -t 7600` - Time limit (in minutes, ~5 days)

### 4. Create Required Directories

On the cluster:
```bash
cd /path/to/scripts/
mkdir -p err out logs
mkdir -p VAR_ruben/hyperparameter_tuning/individual_jobs
```

### 5. Submit the Array Job

```bash
sbatch --array=1-3125 slurm_hyperparameter_tuning.sh
```

This will submit 3125 jobs (one for each priority configuration).

### 6. Monitor Progress

Check job status:
```bash
squeue -u mabavisani
```

Check how many jobs have completed:
```bash
ls VAR_ruben/hyperparameter_tuning/individual_jobs/*.csv | wc -l
```

Check for errors:
```bash
tail err/error*.err
tail out/out*.out
```

### 7. Collect Results (After Jobs Complete)

Once most/all jobs are complete:
```bash
python VAR_collect_parallel_results.py -n 3 -u 3
```

This will:
- Collect all individual job results
- Combine into a single CSV file
- Show summary statistics
- Report any failed jobs

### 8. Analyze Results

After collection, analyze the combined results:
```bash
# Download results to local machine
scp mabavisani@cluster:/path/to/VAR_ruben/hyperparameter_tuning/priority_tuning_*.csv .

# Analyze locally
python VAR_analyze_hyperparameters.py -f priority_tuning_net3_u3_*_parallel.csv
python VAR_generate_report.py -f priority_tuning_net3_u3_*_parallel.csv
```

## Understanding the Job Mapping

Jobs 1-3125 map to priority configurations as follows:

- **Job 1**: `[1, 1, 1, 1, 1]`
- **Job 2**: `[1, 1, 1, 1, 2]`
- **Job 3**: `[1, 1, 1, 1, 3]`
- ...
- **Job 3125**: `[5, 5, 5, 5, 5]`

The mapping is generated by: `itertools.product([1,2,3,4,5], repeat=5)`

## Troubleshooting

### Issue: Jobs are failing immediately

**Check:**
1. Conda environment exists and is activated:
   ```bash
   conda env list | grep multi_v3
   ```
2. Data files exist:
   ```bash
   ls ~/DataSets_Feedbacks/8_VAR_simulation/net3/u3/txtSTD/
   ```
3. Python script has correct path:
   ```bash
   ls -l VAR_hyperparameter_single_job.py
   ```

### Issue: Jobs are timing out

**Solution:** Increase time limit in SLURM script:
```bash
#SBATCH -t 14400  # 10 days instead of 5
```

### Issue: Out of memory errors

**Solution:** Increase memory in SLURM script:
```bash
#SBATCH --mem=4g  # 4GB instead of 2GB
```

### Issue: Some jobs failed

**Solution:** 
1. Check which jobs failed:
   ```bash
   python VAR_collect_parallel_results.py -n 3 -u 3
   # Look at failed_jobs_*.txt
   ```

2. Resubmit only failed jobs:
   ```bash
   # If jobs 100, 205, 350 failed:
   sbatch --array=100,205,350 slurm_hyperparameter_tuning.sh
   ```

### Issue: Collection says "not enough jobs"

**Solution:** Run collection with lower threshold:
```bash
python VAR_collect_parallel_results.py -n 3 -u 3 --min_jobs 2500
```

Or resubmit failed jobs and collect again.

## Estimating Resource Usage

### Time per job:
- Typical: 1-3 hours per priority configuration (5 batches)
- With parallelization: All 3125 jobs complete in 1-3 hours (if cluster has capacity)

### Total resources needed:
- **CPU**: 3125 CPU-hours (for PNUM=1)
- **Memory**: 2GB per job × 3125 = ~6.5TB total (but distributed across jobs)
- **Disk**: ~50MB per job × 3125 = ~150GB for all results

## Tips for Efficient Execution

1. **Start with a small test**: Before submitting all 3125 jobs, test with a small array:
   ```bash
   sbatch --array=1-10 slurm_hyperparameter_tuning.sh
   ```

2. **Use job dependencies**: If you want automatic collection after jobs finish:
   ```bash
   JOB_ID=$(sbatch --array=1-3125 slurm_hyperparameter_tuning.sh | awk '{print $4}')
   sbatch --dependency=afterok:$JOB_ID collection_job.sh
   ```

3. **Monitor cluster load**: Check cluster status to avoid overwhelming the queue:
   ```bash
   squeue | wc -l  # See how many jobs are queued
   ```

4. **Stagger submission**: If cluster policy limits array size, submit in batches:
   ```bash
   sbatch --array=1-1000 slurm_hyperparameter_tuning.sh
   sbatch --array=1001-2000 slurm_hyperparameter_tuning.sh
   sbatch --array=2001-3125 slurm_hyperparameter_tuning.sh
   ```

## Email Notifications

The SLURM script is configured to send emails:
- `--mail-type=ALL` - Send for begin, end, fail, etc.
- `--mail-user=mabavisani@gsu.edu` - Your email

You'll receive emails when:
- Jobs start
- Jobs complete
- Jobs fail

**Note:** With 3125 jobs, you'll get many emails! Consider changing to:
```bash
#SBATCH --mail-type=END,FAIL  # Only end and failures
```

Or create a filter rule in your email to organize these notifications.
